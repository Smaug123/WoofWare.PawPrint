# Property-Based Tests for Type Identity Refactor

## Context

The `refactor-to-type-identity` branch implements phases 0-4 of the type identity refactor plan. The review identified that property-based tests are still missing, but it also identified that the original draft for those tests put too much weight on random *single-assembly* shapes.

That is useful, but it is not where most of the risk lives.

The highest-risk logic in the refactor is:

- cross-assembly resolution through `TypeRefResolutionScope.Assembly`
- nested resolution through `TypeRefResolutionScope.TypeRef`
- exported types and forwarders
- concrete-type uniqueness under generics

Those behaviors live primarily in:

- `WoofWare.PawPrint.Domain/Assembly.fs`
- `WoofWare.PawPrint.Domain/TypeConcretisation.fs`

The existing focused tests in `WoofWare.PawPrint.Test/TestTypeResolution.fs` already cover several of those scenarios with hand-picked examples. The property suite should *complement* those tests, not replace them.

## What This Plan Is For

This plan aims to add mechanical, generated coverage for the invariants introduced by the type-identity refactor:

- handle-based identity roundtrips within an assembly
- nested-type identity and full-name recovery
- scope-aware cross-assembly resolution
- forwarding transparency
- concrete-type uniqueness by resolved nominal identity plus generic arguments

It does **not** attempt to give comprehensive random coverage for the whole type system. In particular, this plan does not try to randomly cover:

- `ModuleRef` support
- arbitrary `TypeSpec` forms
- arrays, pointers, byrefs, pinned, function pointers
- interface implementation metadata
- full inheritance/dispatch/field-layout behavior

Those remain the job of focused tests until there is a reason to broaden the generator.

## Design Principles

### 1. Split the property suite by risk surface

There should be three distinct strata:

1. single-assembly smoke properties
2. assembly-graph resolution properties
3. generic concretisation properties

The original draft treated (1) as the primary deliverable. That is backwards. It should be the cheapest layer, not the most important layer.

### 2. Keep the model small, but be honest about duplication

The model should describe *what to compile* and *which references should exist*. It should not mirror domain-layer output types like:

- `ResolvedTypeIdentity`
- `TypeRef`
- `ExportedType`
- `TypeDefn`
- `ConcreteTypeHandle`

That said, some duplication is unavoidable and acceptable. The generator *will* duplicate the part of the world that matters for these tests:

- namespaces
- type names
- nested-type structure
- generic arity
- selected cross-assembly reference edges

This is still much smaller than the runtime model, and it is the right level of duplication for an independent oracle.

### 3. Do not use the API under test as the oracle

The original draft proposed finding type handles for P3 by walking `TryGetTopLevelTypeDef` and `TryGetNestedTypeDef`. That weakens the tests because those are themselves part of what is being validated.

For generated properties, prefer:

- scanning `assy.TypeDefs`
- reconstructing parent chains from `DeclaringType`
- matching against the expected path from the model

Lookup helpers can still be exercised explicitly, but they should not also be the source of truth for the expected answer.

### 4. Preserve the focused example tests

`TestTypeResolution.fs` already covers important, subtle cases:

- nested `TypeRef` parent chains
- collisions between top-level and nested simple names
- collisions between equal child names under different parents
- top-level forwarders
- synthesized nested exported-type parent chains
- explicit `ModuleRef` failure
- generic concretisation uniqueness/idempotence examples

Those tests should remain. Property tests should broaden coverage around those themes.

## Approach

### 1. Add FsCheck dependency

Add `FsCheck` and `FsCheck.NUnit` to `WoofWare.PawPrint.Test.fsproj`.

**File:** `WoofWare.PawPrint.Test/WoofWare.PawPrint.Test.fsproj`

### 2. Extract shared test helpers

The new property modules will need the same helpers that currently live privately inside `TestTypeResolution.fs`:

- compile a DLL with Roslyn
- load a `DumpedAssembly` from emitted bytes
- build `loadedAssemblies`
- create a minimal concretisation context

Extract those into a small helper module rather than duplicating them across files.

Suggested file:

- `WoofWare.PawPrint.Test/TypeIdentityTestHelpers.fs`

### 3. Define a small single-assembly shape model

This is for smoke properties only.

Suggested file:

- `WoofWare.PawPrint.Test/AssemblyShape.fs`

Suggested model:

```fsharp
type TypeShape = {
    Name : string
    GenericArity : int
    NestedTypes : TypeShape list
}

type NamespaceShape = {
    Name : string
    Types : TypeShape list
}

type AssemblyShape = {
    Name : string
    Namespaces : NamespaceShape list
}
```

Constraints enforced by the generator:

- names are valid C# identifiers drawn from a deliberately small pool
- `(Name, GenericArity)` is unique within each sibling scope
- generic arity is small, e.g. 0-3
- nesting depth is small, e.g. 0-3
- the generator deliberately biases towards collisions:
  - same child names under different parents
  - top-level and nested types sharing a simple name

This model is intentionally limited. It is only meant to drive parser/index/full-name smoke checks.

### 4. Define an assembly-graph model

This is the primary property-testing layer.

Suggested file:

- `WoofWare.PawPrint.Test/AssemblyGraphShape.fs`

This model should generate:

- a defining assembly shape
- a consumer assembly that references selected generated types from the defining assembly
- optionally, a forwarder assembly between the consumer and defining assembly

The model does not need to be fully general. It only needs enough structure to produce the metadata patterns we care about:

- top-level reference in another assembly
- nested reference in another assembly
- nested reference that compiles to a `TypeRefResolutionScope.TypeRef` parent chain
- top-level forwarding
- nested forwarding through an exported parent chain

For nested forwarded exports, generating the entire metadata shape from C# is likely awkward or impossible. Follow the existing pragmatic approach from `TestTypeResolution.fs`: compile the easy part with Roslyn, then synthesize the nested `ExportedType` parent chain in memory where needed.

### 5. Keep generic concretisation as a scenario-specific generator

Do **not** try to force every generic case through the general assembly-shape generator.

The generic bugs are about:

- uniqueness by `ResolvedTypeIdentity + generic args`
- nested generic context propagation

Those are easier and clearer to test with a small scenario generator that emits a few structured templates, for example:

- `Box<T>`
- `Outer<T>.Inner`
- two nominally distinct `Box<T>` definitions with the same simple name in different places

This keeps the generator small and the oracle obvious.

### 6. Write renderers

We need renderers for:

- single-assembly shapes
- assembly-graph scenarios
- generic scenarios

Rendering requirements:

- use metadata names with arity where needed when computing expectations
- emit consumer members that force Roslyn to generate the relevant `TypeRef` rows
- keep bodies simple; fields are often enough

### 7. Write oracle helpers

The oracle layer should compute expected answers from the shape model without using the resolution helpers under test as its own source of truth.

Suggested helpers:

```fsharp
val allTypePaths : AssemblyShape -> TypePath list
val metadataName : TypeSegment -> string
val expectedFullName : TypePath -> string
val findTypeDefByPath : DumpedAssembly -> TypePath -> TypeInfo<GenericParamFromMetadata, TypeDefn>
```

`findTypeDefByPath` should be implemented by scanning `assy.TypeDefs` and reconstructing each type's declaring-type chain from handles, not by calling `TryGetTopLevelTypeDef` / `TryGetNestedTypeDef`.

## Properties

### A. Single-Assembly Smoke Properties

These are useful, but they are not the main event.

**A1: Type-definition partition is complete and disjoint.**

For every `TypeDefinitionHandle` in `assy.TypeDefs`, exactly one of the following is true:

- the type is top-level and is present in the top-level lookup
- the type is nested and is present in the nested lookup

This validates the indexing split in `DumpedAssembly`.

**A2: Identity-to-definition roundtrip.**

For every handle `h` in `assy.TypeDefs`:

`resolveTypeIdentityDefinition assy (ResolvedTypeIdentity.ofTypeDefinition assy.Name h)`

returns a `TypeInfo` whose `.TypeDefHandle = h`.

**A3: Full name matches the model path.**

For every generated type path, find the corresponding `TypeDefinitionHandle` independently, then assert:

`Assembly.fullName assy (ResolvedTypeIdentity.ofTypeDefinition assy.Name h)`

matches the oracle's expected full name.

Be explicit that generic definitions use metadata names, e.g. `Outer`1.Inner`.

**A4: Full names are injective within an assembly.**

Distinct `TypeDefinitionHandle`s in a generated assembly must yield distinct `Assembly.fullName` values.

### B. Assembly-Graph Resolution Properties

These are the highest-priority properties.

**B1: Cross-assembly top-level resolution is idempotent.**

For a generated consumer reference to a generated top-level type, resolving the same `TypeRef` twice yields the same `ResolvedTypeIdentity`.

**B2: Nested `TypeRef` parent-chain resolution is correct.**

For a generated consumer reference to a nested type, the resolved identity points at the defining assembly's matching nested type, even in the presence of same-name collisions elsewhere.

Bias the generator towards:

- `Inner` appearing under multiple different parents
- top-level `Inner` and nested `Inner` both existing

**B3: Direct resolution and forwarder resolution agree.**

For a generated forwarded top-level type, resolving through the forwarder yields the same full name and resolved identity as direct lookup in the target assembly.

**B4: Nested forwarded resolution preserves the exported parent chain.**

For generated nested forwarded scenarios that use a synthesized exported parent chain, resolution through the forwarder yields the same identity/full name as direct lookup in the target assembly.

**B5: Resolved identity always points at a valid definition in the returned assembly.**

Whenever `resolveTypeRef` or `resolveTypeFromExport` returns `Resolved (assy, identity, typeInfo)`, then:

- `identity.Assembly.FullName = assy.Name.FullName`
- `resolveTypeIdentityDefinition assy identity` returns the same `TypeDefHandle`

### C. Generic Concretisation Properties

These are the highest-priority properties after assembly-graph resolution.

**C1: Non-generic type-definition concretisation is idempotent.**

For every generated non-generic definition, `concretizeTypeDefinition` called twice yields the same `ConcreteTypeHandle`.

**C2: Same nominal identity plus same generic arguments is idempotent.**

For a generated generic scenario, concretising the same `TypeDefn.GenericInstantiation` twice yields the same handle.

**C3: Same nominal identity plus different generic arguments yields distinct handles.**

This is the property that actually tests the `ResolvedTypeIdentity + generics` uniqueness key.

**C4: Distinct nominal identities plus the same generic arguments yield distinct handles.**

Generate two distinct nominal types with the same simple name shape, then instantiate both with the same argument list and assert the concrete handles differ.

**C5: Enclosing generic arguments propagate into nested concretisation.**

Generate scenarios like `Outer<T>.Inner` whose field or signature mentions `T`, concretise `Inner` with different outer arguments, and assert:

- repeated concretisation with the same outer argument is idempotent
- changing the outer argument changes the concretised nested type
- the stored concrete generics reflect the chosen argument

## What Remains Example-Based

Keep the focused tests in `TestTypeResolution.fs` for scenarios that are awkward to generate or where the exact failure mode matters:

- explicit `ModuleRef` failure messages
- lazy-load sequencing / specific loader call ordering
- any hand-built metadata that is easier to synthesize directly than to generate through Roslyn

The property suite should expand coverage around those scenarios, not delete them.
