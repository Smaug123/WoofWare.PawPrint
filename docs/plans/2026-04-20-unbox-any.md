# Implement `unbox.any` to unblock CastClassBoxing and CastClassEnum tests

## Context

Two existing C# test files — `WoofWare.PawPrint.Test/sourcesPure/CastClassBoxing.cs` and `.../CastClassEnum.cs` — exercise the `box` → `castclass` → `unbox.any` round trip. `box` and `castclass` were fully implemented, but `unbox.any` was a `failwith` stub. Both tests were parked in the `unimplemented` set in `WoofWare.PawPrint.Test/TestPureCases.fs`.

Goal: implement enough of ECMA-335 III.4.33 (`unbox.any`) to make those two tests pass, plus add tests covering primitive unwrapping paths. `Nullable<T>` handling is deferred to a follow-up PR.

## What was implemented

### Scope

- Value-type exact-match path (pushes `UserDefinedValueType` for user-defined structs/enums)
- Primitive unwrapping: boxed primitives (Int32, Int64, Bool, Char, Float, etc.) are unwrapped from their single-field struct representation back to the appropriate `EvalStackValue` variant, so that arithmetic IL opcodes (Add, etc.) can consume them
- IntPtr/UIntPtr special-case: these are value types whose `cliTypeZeroOfHandle` returns `CliType.ValueType`, but the eval stack requires `NativeInt`; detected via namespace/name matching and unwrapped alongside other primitives. This is spec-mandated: in the real CLR, [`CEEInfo::asCorInfoType`](https://github.com/dotnet/dotnet/blob/10060d128e3f470e77265f8490f5e4f72dae738e/src/runtime/src/coreclr/vm/jitinterface.cpp#L314-L410) maps `ELEMENT_TYPE_I` to [`CORINFO_TYPE_NATIVEINT` (line 381)](https://github.com/dotnet/dotnet/blob/10060d128e3f470e77265f8490f5e4f72dae738e/src/runtime/src/coreclr/vm/jitinterface.cpp#L381)
- Null → `NullReferenceException` for non-nullable value-type targets
- Array heap objects with value-type target → `InvalidCastException` (not an interpreter abort)
- Reference-type delegation (inlined castclass-style logic: null passthrough, assignability check, `InvalidCastException` on failure)
- `Nullable<T>` → explicit `failwith "TODO: Unbox_Any for Nullable<T> unimplemented"`. Nullable special-casing is spec-mandated (ECMA-335 III.4.16): boxing `Nullable<T>` produces `null` or `boxed<T>`, not `boxed<Nullable<T>>`. The real CLR uses a [dedicated `MethodTable` flag](https://github.com/dotnet/dotnet/blob/10060d128e3f470e77265f8490f5e4f72dae738e/src/runtime/src/coreclr/vm/methodtable.h#L2780) set [during class loading](https://github.com/dotnet/dotnet/blob/10060d128e3f470e77265f8490f5e4f72dae738e/src/runtime/src/coreclr/vm/methodtablebuilder.cpp#L10373) rather than name-based detection

### Key implementation detail: primitive unwrapping via `cliTypeZeroOfHandle`

The `Box` path stores primitives as single-field `CliValueType` structs on the managed heap. On `unbox.any`, we must reverse this: if the target type's zero representation (`cliTypeZeroOfHandle`) is a primitive CliType (not `CliType.ValueType`), or the target is `IntPtr`/`UIntPtr`, we unwrap using `CliValueType.TryExactlyOneField` + `EvalStackValue.ofCliType`. Otherwise we push `UserDefinedValueType` directly.

### Why exact-type match (not assignability) for value types

ECMA-335 III.4.33 specifies type-identity for the value-type branch. `System.Enum` is a reference type, so `unbox.any` on it takes the reference-type path (which uses assignability). If we allowed assignability on the value-type branch, we'd accept e.g. a boxed `Color` unboxed as some base `struct` it nominally inherits from — contrary to spec and real CLR behaviour.

## Test files added

All in `WoofWare.PawPrint.Test/sourcesPure/` (auto-discovered):

- `UnboxAnyPrimitiveArithmetic.cs` — `(int)boxed + 1` where boxed is `-1`, confirming Int32 unwrap works with arithmetic
- `UnboxAnyInt64Arithmetic.cs` — `(long)boxed + 1L` where boxed is `41L`, confirming Int64 unwrap
- `UnboxAnyBoolBranch.cs` — `(bool)boxed` in a conditional, confirming Bool unwrap
- `UnboxAnyNintArithmetic.cs` — `(nint)boxed + 1` where boxed is `(nint)41`, confirming IntPtr unwrap

### Error-path tests deferred

The original plan called for `UnboxAnyNullValueType.cs` (null → NullReferenceException) and `UnboxAnyWrongType.cs` (wrong type → InvalidCastException) using try/catch. These were attempted but hit the Monitor/SR.GetResourceString/RuntimeTypeHandle::GetAssembly chain during exception construction — the same chain that blocks `CastClassInvalid.cs`. The error paths are implemented and will be testable once that chain is unblocked.
