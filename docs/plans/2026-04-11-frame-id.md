# Plan: Replace raw `int` frame indices with abstract `FrameId` type

## Goal

Replace raw frame indices with `FrameId` and, more importantly, create a single resolution seam for frame lookup and mutation. All conversion from `FrameId` to a concrete `MethodState` should live in `ThreadState` and `IlMachineState`, so a later "is this frame still live?" check can be added without another repo-wide refactor.

## Context

`ManagedPointerSource.LocalVariable` and `.Argument` currently store a raw `int` index into `ThreadState.MethodStates`. `ThreadState.ActiveMethodState` and `MethodReturnState.JumpTo` are also raw `int`s.

A plain DU wrapper is not enough if callers immediately destructure it everywhere. That would give nominal typing, but no abstraction boundary. If we want future frame-liveness validation, frame resolution must be centralized now.

This should be a mostly mechanical, compiler-guided refactor with no intended runtime behaviour change. It is still a source-breaking API change within the library.

## Design Rule

- `FrameId` may be stored anywhere.
- Resolving a `FrameId` to a `MethodState`, mutating a frame, appending a frame, and switching the active frame should happen only through `ThreadState` and `IlMachineState`.
- Do not add scattered `let (FrameId idx) = ...` unwrapping at call sites. Keep unwrapping in one place so future liveness validation has a real hook.

## Step 1: Introduce `FrameId` in `AbstractMachineDomain.fs`

Add alongside `ThreadId` and `ManagedHeapAddress`:

```fsharp
type FrameId =
    | FrameId of int

    override this.ToString () =
        match this with
        | FrameId.FrameId i -> $"<frame #%i{i}>"
```

If convenient, making the case private is a bonus, but not required for this change. The important invariant is centralized resolution, not constructor privacy.

## Step 2: Retype frame references

Update the places that currently treat frame identity as a raw `int`:

- `ManagedPointerSource.LocalVariable of ThreadId * FrameId * uint16`
- `ManagedPointerSource.Argument of ThreadId * FrameId * uint16`
- `MethodReturnState.JumpTo : FrameId`
- `ThreadState.ActiveMethodState : FrameId`
- Bootstrapping sites in `ThreadState.New` and `Program.fs` use `FrameId 0` or a helper

Update `ManagedPointerSource.ToString` to print the frame with `%O`.

## Step 3: Add frame-resolution/update primitives to `ThreadState`

`ThreadState` should own within-thread frame access. Add helpers with names to suit the local style; the important operations are:

- `getFrame : FrameId -> ThreadState -> MethodState`
- `setFrame : FrameId -> MethodState -> ThreadState -> ThreadState`
- `mapFrame : FrameId -> (MethodState -> MethodState) -> ThreadState -> ThreadState`
- `appendFrame : MethodState -> ThreadState -> FrameId * ThreadState`
- `setActiveFrame : FrameId -> ThreadState -> ThreadState`

`member this.MethodState` should resolve `ActiveMethodState` via this API rather than indexing directly.

All `FrameId` to `int` extraction belongs inside these helpers. This is the place where later liveness validation can fail fast if a frame is dead.

Once these helpers exist, rewrite existing `ThreadState` operations in terms of them:

- `peekEvalStack`
- `popFromEvalStack`
- `pushToEvalStack`
- `pushToEvalStack'`
- `jumpProgramCounter`
- `advanceProgramCounter`
- `loadArgument`
- `setLocalVariable`

No caller outside `ThreadState` should need to index `MethodStates` by `FrameId`.

## Step 4: Add cross-thread resolution helpers to `IlMachineState`

`ManagedPointerSource` carries both `ThreadId` and `FrameId`, so `IlMachineState` should own cross-thread resolution. The minimum useful helpers are:

- `getFrame : ThreadId -> FrameId -> IlMachineState -> MethodState`
- `setFrame : ThreadId -> FrameId -> MethodState -> IlMachineState -> IlMachineState`
- `mapFrame : ThreadId -> FrameId -> (MethodState -> MethodState) -> IlMachineState -> IlMachineState`

Then retype:

- `getLocalVariable` to take `FrameId`
- `setLocalVariable` to take `FrameId`

`dereferencePointer`, `getFieldValue`, and `setFieldValue` should go through these helpers rather than indexing `MethodStates` directly.

This gives one seam for future "frame id exists but is not live" validation.

## Step 5: Rewrite call sites to use the new seam

The remaining work is mechanical, but the rule is:

- Active-frame operations use `ThreadState.MethodState`, `ThreadState.mapFrame`, or `IlMachineState.mapFrame`
- Cross-frame and pointer-based operations use `IlMachineState.getFrame`, `getLocalVariable`, or `setLocalVariable`
- No module outside `ThreadState.fs` should destructure `FrameId` or index `MethodStates` with it

Expected call-site updates:

- `IlMachineState.fs`
  - `returnStackFrame` uses `ThreadState.setActiveFrame` and `ThreadState.getFrame`
  - `popFromStackToLocalVariable` uses `ThreadState.MethodState` and `ThreadState.mapFrame`
  - `getLocalVariable`, `setLocalVariable`, `dereferencePointer`, `getFieldValue`, and `setFieldValue` retype to `FrameId`
- `IlMachineStateExecution.fs`
  - use `threadState.MethodState` instead of `MethodStates.[ActiveMethodState]`
  - when calling a method, use `appendFrame` to allocate the callee frame id instead of building it from `MethodStates.Length`
  - use `setActiveFrame` when switching to callee or caller
- `UnaryConstIlOp.fs`
  - replace direct active-frame indexing and `SetItem` with `ThreadState.MethodState` and `ThreadState.mapFrame`
  - `ManagedPointerSource.LocalVariable` and `.Argument` construction becomes type-driven once `ActiveMethodState` is `FrameId`
- `NullaryIlOp.fs`
  - same active-frame rewrite pattern
- `UnaryMetadataIlOp.fs`
  - same active-frame rewrite pattern
  - also update the direct frame dereferences in the `ldfld`-style paths that currently read `MethodStates.[methodFrame]` directly; the earlier checklist missed these
- `ExternImplementations/System.Threading.Monitor.fs`
  - resolve `methodFrame` via `IlMachineState` or `ThreadState` helpers instead of direct indexing
- `Program.fs`
  - initial main frame becomes `FrameId 0`

## Step 6: Explicit Non-Goals

- No liveness tracking yet. This refactor only creates the abstraction boundary needed for it.
- No generation or version counter yet.
- No intended runtime semantics change, beyond the library API becoming more strongly typed.

## Why This Is The Right Shape

A raw `int` -> `FrameId` rename gives type safety, but it does not create a place to enforce "this frame is still valid". Centralizing resolution now keeps the change mechanical today and makes the later liveness-checking change small.
